FLUJO CORRECTO DE CREACIÓN DE ORDEN

Validar disponibilidad de producto o variante.

Validar método de pago y tipo de servicio.

INSERT en orders con total = 0.

Por cada item:

Determinar precio_unitario

Calcular total_item = cantidad * precio_unitario

INSERT en order_items

INSERT modificadores si existen

SUMAR todos los order_items.total.

UPDATE orders.total.

Cambiar estado_pedido a 'confirmado'.

Si es POS offline:

INSERT en operation_queue.

============================================================
6. FLUJO TRANSACCIONAL DE CREACIÓN DE ORDEN
============================================================
6.1. Descripción General

El proceso de creación de una orden debe ejecutarse dentro de una transacción atómica para garantizar:

Consistencia de datos

Integridad referencial

No generación de órdenes incompletas

Manejo adecuado de errores (ROLLBACK)

Todas las operaciones deben ejecutarse en el servidor.

6.2. Validaciones Previas (Servidor)

Antes de iniciar la transacción se deben validar las siguientes reglas de negocio:

Verificar disponibilidad del producto o variante:

Campo disponible = true

Validar stock si el sistema lo maneja

Validar método de pago:

Debe existir en metodos_pago

Debe estar activo si se maneja estado

Validar tipo de servicio:

Debe existir en tipos_servicio

Validar turno activo si aplica.

Si alguna validación falla:

No iniciar transacción

Retornar error controlado

6.3. Secuencia Operacional
Paso 1: Iniciar Transacción
BEGIN;
Paso 2: Crear Orden (Estado Inicial)

Se inserta la orden con total provisional igual a 0.

INSERT INTO public.orders (
  cliente_id,
  fecha_creacion,
  fecha_cierre,
  total,
  propina,
  estado_pedido,
  estado_pago,
  metodo_pago_id,
  tipo_servicio_id,
  turno_id
)
VALUES (
  :cliente_id,
  NOW(),
  NULL,
  0::numeric,
  COALESCE(:propina, 0)::numeric,
  COALESCE(:estado_pedido, 'pendiente')::varchar,
  COALESCE(:estado_pago, 'pendiente')::varchar,
  :metodo_pago_id,
  :tipo_servicio_id,
  :turno_id
)
RETURNING id AS order_id;

Resultado:

Se obtiene order_id generado automáticamente.

Paso 3: Procesamiento de Ítems del Carrito

Para cada ítem recibido en el carrito:

3.1 Determinar Precio Unitario

Regla:

Si existe variante → usar producto_variantes.precio

Si no → usar productos.precio

Consulta sugerida:

SELECT precio
FROM public.producto_variantes
WHERE id = :variante_id
  AND disponible = true;

O:

SELECT precio
FROM public.productos
WHERE id = :producto_id
  AND disponible = true;
3.2 Insertar Ítem en order_items
INSERT INTO public.order_items (
  order_id,
  producto_id,
  nombre_producto,
  cantidad,
  precio_unitario,
  total
)
VALUES (
  :order_id,
  :producto_id,
  :nombre_producto,
  :cantidad,
  :precio_unitario::numeric,
  (:cantidad * :precio_unitario)::numeric
)
RETURNING id AS order_item_id;

Resultado:

Se obtiene order_item_id

3.3 Insertar Modificadores (Opcional)

Si el producto contiene modificadores:

INSERT INTO public.order_item_modificadores (
  order_item_id,
  nombre_modificador,
  cantidad,
  precio_unitario
)
VALUES (
  :order_item_id,
  :nombre_mod,
  :mod_cantidad,
  :mod_precio_unitario
);

Este paso puede repetirse múltiples veces por item.

Paso 4: Calcular Total de la Orden

Se suman los totales de los items:

SELECT COALESCE(SUM(total), 0)::numeric AS items_total
FROM public.order_items
WHERE order_id = :order_id;
Paso 5: Actualizar Orden con Total Final
UPDATE public.orders
SET total = :items_total + COALESCE(:propina, 0)::numeric,
    estado_pedido = COALESCE(:nuevo_estado_pedido, estado_pedido),
    fecha_cierre = CASE 
        WHEN :cerrar_ahora = true 
        THEN NOW() 
        ELSE fecha_cierre 
    END
WHERE id = :order_id;

Reglas:

El total final incluye propina.

Puede incluir impuestos/envío según reglas del negocio.

Puede cerrarse inmediatamente o quedar abierta.

Paso 6: Encolar Operación (Opcional – Offline / Async)

Si el sistema maneja sincronización o procesamiento adicional:

INSERT INTO public.operation_queue (
  uuid,
  type,
  payload,
  status,
  created_at
)
VALUES (
  gen_random_uuid()::text,
  'process_order',
  :payload_json::text,
  'pending',
  extract(epoch FROM now())::int
);

Uso típico:

Notificaciones

Facturación electrónica

Sincronización con servidor central

Paso 7: Confirmar Transacción
COMMIT;

En caso de error:

ROLLBACK;
6.4. Propiedades Técnicas del Flujo

✔ Atomicidad
✔ Consistencia
✔ Integridad referencial
✔ Reproducible
✔ Compatible con arquitectura offline-first

6.5. Buenas Prácticas Recomendadas

Ejecutar siempre dentro de una transacción.

Usar parámetros preparados (evitar SQL injection).

No calcular totales en frontend como fuente de verdad.

Usar índices en:

order_items.order_id

orders.fecha_creacion

Validar estado del producto antes del insert.

Registrar auditoría si el sistema lo requiere.
